https概述

Author: Cynyard Qiu Date: December 8, 2021  20:00:00 Category: Network

概念

http over SSL，而TLS是在SSL的基础上改进的协议

整体流程

目标

  • 信息加密传播
  • 校验机制，信息被篡改后通信双方会发现
  • 配置身份证书，防止身份被冒充

    流程

建立TCP连接后的流程

  • 客户端向服务端索要并验证公钥
  • 双方协商生成对话密钥
  • 双方采用对话密钥进行加密通信

对于前两部，叫做握手阶段，设计四次通信，这四次通信都是明文的

握手阶段

图片是使用wireshark抓包Bing.com ： ip.addr == xxx and tls

客户端请求

TLSv1.2

浏览器向服务器发出加密通信的请求，Client Hello，并且向服务器提供以下信息

  • 支持的协议版本，TLS x.x
  • 客户端生成的随机数Random1
  • 客户端支持的加密方法
  • 客户端支持的压缩方法

服务器回应

TLSv1.2

Server Hello，确认使用的加密通信协议版本，如果双方支持的版本不一致，就关闭加密通信，一致的话发送

  • 服务器生成的随机数Random2
  • 确认使用的加密方法
  • 服务器由CA颁发的证书以及公钥

有些时候需要确认客户端的身份，就会要求客户端提供客户端证书

客户端回应

TLSv1.2 这里没找到PreMaster Key，采用DH算法，Premaster secret不需要传递，双方只要交换各自的参数，就可以算出这个随机数。

Client Key ExChange
验证服务器证书，如果不是客户端可信的机构或者已经过期或者证书中的域名与实际域名不一致，就会向访问者显示一个警告
如果证书没有问题，取出证书中服务器的公钥，生成随机数Random3，然后发送

  • 使用公钥加密Random3，也叫Pre-Master Key
  • 编码改变通知，就用你说的这个加密方法
  • 客户端握手结束通知，也是前面发送的所有内容的hash，用来供服务器教研

服务端最后回应

TLSv1.2 使用了Session Ticket

Change Cipher Spec Finished
在客户端回应过后，服务器在本地使用私钥解密Pre-Master Key，至此，客户端和服务器同时拥有了三个随机数
然后双方就用事先商定的加密方法，通过密钥生成器各自生成本次会话所用的对话密钥（后面的对称加密使用），最后回应

  • session ticket 下一次请求https的时候，可以使用这个session ticket，后续的https请求可以节省很多资源
  • 告诉客户端以后的通讯都用对称密钥的方式进行
  • 服务端握手结束，也是前面发送的所有内容的hash，用来供客户端校验

至此，握手阶段全部结束，然后客户端与服务器进入加密通信

通信阶段

完全使用普通的HTTP协议，只不过使用会话密钥加密内容

证书

CA ： 证书授权中心，CA本身是受信任的，CA给信任的申请对象颁发证书
证书要买啊，自己的服务端不想交钱咋办，自己制作证书，然后把证书的公钥放在服务器，浏览器使用自己的证书公钥就可以解密了
为了防止公钥证书泄露，可以使用数字签名认证算法（TODO）

Session ID和Session Ticket

每一次对话都有一个编号（session ID）。如果对话中断，下次重连的时候，只要客户端给出这个编号，且服务器有这个编号的记录，双方就可以重新使用已有的”对话密钥”，不再进行握手阶段剩余的步骤重新生成一把。
session ID是目前所有浏览器都支持的方法，但是它的缺点在于session ID往往只保留在一台服务器上。所以，如果客户端的请求发到另一台服务器，就无法恢复对话
session ticket就是为了解决这个问题而诞生的
客户端不再发送session ID，而是发送一个服务器在上一次对话中发送过来的session ticket。这个session ticket是加密的，只有服务器才能解密，其中包括本次对话的主要信息，比如对话密钥和加密方法。
